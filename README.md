##  Задачи проекта

1.  **Архитектура:** Отработка микросервисного подхода с API Gateway, логическим разделением сервисов и контейнеризацией через Docker.
2.  **Безопасность:** - аутентификации через JWT с централизованной валидацией на Gateway.
3.  **Data Management:** Работа с MySQL, изучение организации данных в распределенной системе. 


##  Архитектура

### Аутентификация и передача контекста

1.  **Валидация на Gateway:** API Gateway проверяет подпись JWT и срок действия токена.
2.  **Извлечение Claims:** Из валидного токена извлекаются данные пользователя (`user_id`, `email`, `username`).
3.  **Проброс заголовков:** Gateway добавляет данные в HTTP-заголовки для внутренних сервисов:
``` 
		if userID, ok := claims["user_id"].(float64); ok {
			r.Header.Set("X-User-ID", fmt.Sprintf("%.0f", userID))
		}1
```

| Claim в JWT | Заголовок | Описание |
|-------------|-----------|----------|
| `user_id` | `X-User-ID` | Уникальный ID пользователя |
| `email` | `X-User-Email` | Email пользователя |
| `username` | `X-User-Username` | Логин пользователя |

4.  **Обработка в сервисе:** Микросервисы получают контекст из заголовков и используют его для бизнес-логики, не занимаясь парсингом токена.

> !!! **Важно:** Внутренние сервисы доверяют заголовкам `X-User-*` только потому, что они добавляются доверенным Gateway. Прямые запросы извне на порты сервисов заблокированы.

###  Стратегия работы с данными

На текущем этапе используется **единый экземпляр MySQL** с логическим разделением таблиц по доменам:

| Сервис | Таблицы |
|--------|---------|
| `auth` | `users`, |
| `user` | `posts`, `comments`, `likes` |
| `chat` | `messages`, `dialogs` 🚧 (в разработке)| 

**План развития:** Переход на паттерн *Database-per-Service* для полной изоляции данных.
